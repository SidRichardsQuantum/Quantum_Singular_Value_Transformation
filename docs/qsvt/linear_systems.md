# Linear Systems

This page summarizes the theory behind the linear-system paths in the package:
QSVT-style inverse polynomials, small solver comparisons, and finite HHL circuit
execution. For the concise per-workflow page, see
[Linear system workflow](workflow_linear_system.md). For comparison rows across
dense, iterative, QSVT-style, and HHL paths, see
[Linear system comparison workflow](workflow_linear_system_comparison.md).

The common mathematical target is a positive-definite system

$$
A x = b,
$$

where the desired operation is the inverse action \(x = A^{-1} b\). Quantum
linear-system algorithms usually represent the normalized vector proportional
to \(A^{-1}|b\rangle\), rather than returning every entry of \(x\) directly.

## QSVT inverse polynomials

QSVT applies bounded polynomials to singular values or eigenvalues of a
block-encoded operator. For a positive-definite Hermitian matrix, the package
uses a normalized positive spectrum

$$
\lambda(A / \alpha) \subseteq [\gamma, 1],
$$

where \(\alpha\) is a positive scaling factor and \(\gamma\) is the smallest
scaled eigenvalue. The ideal inverse function \(1/x\) is too large near zero,
so the implemented target is the bounded function

$$
P(x) \approx \frac{\gamma}{x}
\quad \text{for } x \in [\gamma, 1].
$$

After applying \(P(A / \alpha)\) to \(b\), the package rescales by
\(\gamma \alpha\) to approximate \(A^{-1}b\). This is the core idea behind
`qsvt.algorithms.linear_system_workflow`; see
[Linear system workflow](workflow_linear_system.md) for the target,
implementation, diagnostics, and API summary.

The approximation becomes harder as \(\gamma\) gets smaller. Since
\(\gamma^{-1}\) tracks conditioning after normalization, ill-conditioned
systems require higher polynomial degree and tend to have larger residuals at
fixed degree.

## Solver comparison workflow

`qsvt.algorithms.linear_system_comparison_workflow` compares several finite
small-matrix paths for the same problem:

- dense direct solve,
- optional conjugate-gradient solve,
- QSVT-style polynomial inverse action,
- optional PennyLane QSVT matrix check.

The comparison rows are numerical diagnostics, not runtime claims about a full
quantum linear-system solver. They are useful for asking whether the polynomial
inverse row approximates the dense reference on the finite instance and how the
degree, conditioning, residual, and optional QSVT check relate.

See [Linear system comparison workflow](workflow_linear_system_comparison.md).

## Experimental HHL comparison

The package also includes `qsvt.comparisons.execute_hhl_circuit`, a finite
simulator-scale implementation of the HHL circuit pattern. It is an adjacent
experimental comparison and tutorial, not a core QSVT implementation or stable
package milestone. HHL uses quantum phase estimation on a unitary generated by
\(A\), a controlled reciprocal rotation, inverse phase estimation, and ancilla
postselection.

At a high level:

1. prepare the normalized right-hand-side state \(|b\rangle\),
2. run phase estimation for \(e^{iAt}\),
3. rotate an ancilla by an amplitude proportional to \(C / \lambda_j\),
4. uncompute the phase register,
5. postselect the ancilla to obtain a state proportional to \(A^{-1}|b\rangle\).

The reciprocal rotation scale \(C\) must not exceed the occupied lower
eigenvalue bound. Otherwise the controlled rotation would require amplitudes
larger than one.

For dense matrix inputs, the helper materializes explicit time-evolution
unitaries through PennyLane so the circuit can run on `default.qubit`. Callers
can instead provide `controlled_time_evolution` to queue a problem-specific
oracle-compatible evolution block.

## Diagnostics

Linear-system reports focus on:

- residual norm \(\|Ax-b\|\),
- relative solution error against the dense reference,
- condition-number metadata,
- \(\gamma\) and scaled spectral bounds,
- polynomial diagnostics and compatibility metadata,
- optional QSVT or HHL state/circuit diagnostics.

For HHL, the report also records postselection success probability, fidelity or
state error when a classical reference is available, phase-register size,
rotation scale, and resource metadata from PennyLane.

## Scope

The implemented workflows are finite validation paths. They do not provide
production-scale block encodings, scalable Hamiltonian simulation oracles,
state-preparation costs, amplitude amplification, output tomography, hardware
compilation, or fault-tolerant synthesis.

Use the QSVT-style inverse workflows to study polynomial degree and residual
behavior. Use the HHL circuit helper to inspect the circuit mechanics on small
instances. Treat both as explicit finite implementations with clear omitted
layers, not as full asymptotic quantum-runtime benchmarks.
